DevOps Learning Path

Master the tools and practices that bridge development and operations

Module 1: DevOps Fundamentals

What Is DevOps?

DevOps is a set of practices, cultural philosophies, and tools that combines software development (Dev) and IT operations (Ops). It aims to shorten the systems development lifecycle while delivering features, fixes, and updates frequently and reliably. DevOps is not a tool or a job title—it is a culture and methodology.

Core Principles

  • Culture of Collaboration: Break down silos between development, operations, and QA teams. Everyone shares responsibility for the product.
  • Automation: Automate repetitive tasks like testing, deployment, infrastructure provisioning, and monitoring to reduce human error and increase speed.
  • Continuous Improvement: Constantly measure, learn, and improve processes through feedback loops and retrospectives.
  • Infrastructure as Code: Manage infrastructure through version-controlled code rather than manual configuration.
  • Monitoring and Feedback: Continuously monitor applications and infrastructure to detect issues early and gather data for improvement.

The DevOps Lifecycle

The DevOps lifecycle is an infinite loop of interconnected phases that enable rapid, reliable software delivery:

  • Plan: Define features, prioritize work, and track progress using agile methodologies
  • Code: Write and review code collaboratively using version control
  • Build: Compile code, run unit tests, and create deployable artifacts
  • Test: Run automated tests including integration, performance, and security tests
  • Release: Approve and prepare artifacts for deployment to production
  • Deploy: Push changes to production using automated deployment strategies
  • Operate: Manage and maintain production infrastructure and applications
  • Monitor: Collect metrics, logs, and traces to understand system behavior and detect issues

Module 2: Linux & Scripting

Why Linux Matters

Linux powers the vast majority of servers, cloud instances, and containers in the world. Understanding Linux is non-negotiable for any DevOps or cloud engineer. Most production systems, CI/CD agents, and container images run on Linux distributions like Ubuntu, CentOS, Amazon Linux, or Alpine.

Essential Linux Commands

  • Navigation: cd, ls, pwd, find, locate, tree
  • File Operations: cp, mv, rm, mkdir, touch, cat, head, tail, less
  • Text Processing: grep, sed, awk, cut, sort, uniq, wc
  • System Info: top, htop, df, du, free, uname, uptime
  • Networking: curl, wget, ping, netstat, ss, dig, nslookup, ip
  • Process Management: ps, kill, systemctl, journalctl, cron

File Permissions and Ownership

Linux uses a permission system based on three levels: owner, group, and others. Each level can have read (r=4), write (w=2), and execute (x=1) permissions. Understanding permissions is critical for security and proper system administration.

  • chmod 755 file.sh — owner: rwx, group: r-x, others: r-x
  • chown user:group file — change file ownership
  • sudo — execute commands with superuser privileges

Shell Scripting (Bash)

Bash scripting is essential for automating repetitive tasks in DevOps. Every DevOps engineer should be able to write scripts for deployment, log rotation, backup, monitoring checks, and system maintenance.

  • Variables, conditionals (if/else), loops (for, while), and functions
  • Command substitution, piping (|), and redirection (>, >>, 2>)
  • Exit codes and error handling
  • Reading input, processing files, and parsing command output
  • Cron jobs for scheduling automated tasks

Module 3: Git & Version Control

Why Version Control?

Version control systems track every change made to code over time, allowing teams to collaborate without overwriting each other's work, review changes before merging, and roll back to any previous state if something goes wrong. Git is the industry-standard distributed version control system used by virtually every software team.

Core Git Concepts

  • Repository: A directory tracked by Git containing all project files and the complete history of changes
  • Commit: A snapshot of your changes with a descriptive message. Each commit has a unique SHA hash.
  • Branch: An independent line of development. The default branch is usually called main or master.
  • Merge: Combining changes from one branch into another
  • Remote: A hosted copy of the repository (e.g., on GitHub, GitLab, or Bitbucket)
  • Clone: Creating a local copy of a remote repository

Essential Git Commands

  • git init — Initialize a new repository
  • git clone <url> — Clone a remote repository
  • git add . — Stage all changes for commit
  • git commit -m "message" — Create a commit with a message
  • git push origin main — Push local commits to the remote repository
  • git pull — Fetch and merge changes from the remote
  • git branch feature-x — Create a new branch
  • git checkout feature-x — Switch to a branch
  • git merge feature-x — Merge a branch into the current branch
  • git log --oneline — View commit history
  • git stash — Temporarily save uncommitted changes
  • git rebase — Reapply commits on top of another branch

Branching Strategies

  • Git Flow: Uses main, develop, feature/*, release/*, and hotfix/* branches. Best for projects with scheduled releases.
  • GitHub Flow: Simple model with main and short-lived feature branches. Every change goes through a pull request. Best for continuous deployment.
  • Trunk-Based Development: Developers commit directly to main (or short-lived branches). Requires robust automated testing. Used by high-performing teams.

Pull Requests and Code Review

Pull requests (PRs) or merge requests (MRs) are a mechanism for proposing changes, discussing them with teammates, and getting approval before merging into the main branch. Code reviews improve code quality, share knowledge, and catch bugs early. Every DevOps team should enforce code review as a standard practice.

Module 4: Docker

What Are Containers?

Containers are lightweight, standalone, executable packages that include everything needed to run a piece of software: the code, runtime, system tools, libraries, and settings. Unlike virtual machines, containers share the host operating system's kernel, making them much faster to start and more efficient with resources.

Docker is the most popular containerization platform. It standardizes how applications are packaged, distributed, and run, solving the classic "it works on my machine" problem.

Core Docker Concepts

  • Image: A read-only template containing the application and its dependencies. Built from a Dockerfile.
  • Container: A running instance of an image. Containers are ephemeral by default.
  • Dockerfile: A text file with instructions to build a Docker image (FROM, RUN, COPY, CMD, EXPOSE, etc.)
  • Registry: A storage and distribution system for Docker images. Docker Hub is the default public registry.
  • Volume: Persistent storage that survives container restarts and deletions
  • Network: Docker creates isolated networks for container communication

Essential Docker Commands

  • docker build -t myapp . — Build an image from a Dockerfile
  • docker run -d -p 8080:80 myapp — Run a container in detached mode with port mapping
  • docker ps — List running containers
  • docker stop <id> — Stop a running container
  • docker logs <id> — View container logs
  • docker exec -it <id> bash — Open a shell inside a running container
  • docker images — List local images
  • docker push / docker pull — Push and pull images from a registry

Dockerfile Best Practices

  • Use minimal base images (Alpine Linux is only 5MB)
  • Leverage multi-stage builds to keep final images small
  • Order instructions from least to most frequently changing to maximize layer caching
  • Never store secrets in Docker images
  • Use .dockerignore to exclude unnecessary files from the build context
  • Run containers as a non-root user for security

Docker Compose

Docker Compose is a tool for defining and running multi-container applications using a YAML file (docker-compose.yml). It lets you define services, networks, and volumes in a single file and start everything with docker compose up. It is essential for local development environments with multiple interconnected services.

Module 5: Kubernetes

What Is Kubernetes?

Kubernetes (K8s) is an open-source container orchestration platform originally developed by Google. It automates the deployment, scaling, and management of containerized applications. Kubernetes handles load balancing, self-healing, rolling updates, storage orchestration, and secret management, making it the industry standard for running containers in production.

Core Kubernetes Objects

  • Pod: The smallest deployable unit in Kubernetes. A pod contains one or more containers that share networking and storage.
  • Deployment: Manages a set of identical pods and handles rolling updates and rollbacks. The most common way to run stateless applications.
  • Service: Provides a stable network endpoint (IP and DNS name) for accessing a set of pods. Types include ClusterIP, NodePort, and LoadBalancer.
  • ConfigMap: Stores non-sensitive configuration data as key-value pairs, injected into pods as environment variables or files.
  • Secret: Stores sensitive data like passwords, tokens, and keys. Base64 encoded (not encrypted by default).
  • Namespace: Logical partition within a cluster for organizing and isolating resources.
  • Ingress: Manages external HTTP/HTTPS access to services within the cluster, providing routing, SSL termination, and virtual hosting.

Essential kubectl Commands

  • kubectl get pods — List all pods in the current namespace
  • kubectl get svc — List all services
  • kubectl apply -f manifest.yaml — Apply a configuration from a YAML file
  • kubectl describe pod <name> — Show detailed information about a pod
  • kubectl logs <pod> — View pod logs
  • kubectl exec -it <pod> -- bash — Execute a command inside a pod
  • kubectl scale deployment <name> --replicas=5 — Scale a deployment
  • kubectl delete -f manifest.yaml — Delete resources defined in a YAML file

Kubernetes Architecture

  • Control Plane: API Server (cluster gateway), etcd (key-value store for cluster state), Scheduler (assigns pods to nodes), Controller Manager (maintains desired state)
  • Worker Nodes: kubelet (ensures containers are running in pods), kube-proxy (handles networking rules), Container Runtime (runs containers, e.g., containerd)

Helm

Helm is the package manager for Kubernetes. It uses "charts" (pre-configured Kubernetes resource packages) to simplify deploying and managing complex applications. Helm charts are reusable, versioned, and configurable with values files, making it easy to deploy applications like Nginx, Prometheus, or PostgreSQL with a single command.

Module 6: CI/CD

What Is CI/CD?

Continuous Integration (CI) is the practice of automatically building and testing code every time a developer pushes changes to the repository. Continuous Delivery (CD) extends CI by automatically deploying all code changes to a staging or production environment after passing tests. Together, CI/CD pipelines enable teams to release software faster and with greater confidence.

CI/CD Pipeline Stages

  • Source: Triggered when code is pushed or a pull request is created
  • Build: Compile code and create deployable artifacts (Docker images, JARs, binaries)
  • Test: Run unit tests, integration tests, security scans, and code quality checks
  • Deploy to Staging: Deploy to a staging environment for validation
  • Approval (optional): Manual approval gate before production deployment
  • Deploy to Production: Deploy to the production environment using a chosen strategy
  • Post-Deploy: Run smoke tests, monitor for errors, and notify the team

Popular CI/CD Tools

  • GitHub Actions: Native CI/CD built into GitHub. Uses YAML workflow files. Free for public repositories. Excellent for teams already on GitHub.
  • GitLab CI/CD: Built into GitLab with powerful pipeline capabilities. Uses .gitlab-ci.yml. Supports auto-scaling runners.
  • Jenkins: The most widely used open-source automation server. Highly extensible with thousands of plugins. Requires self-hosting and maintenance.
  • AWS CodePipeline: Fully managed CI/CD service that integrates natively with AWS services.
  • Azure DevOps Pipelines: Full-featured CI/CD with both YAML and visual pipeline editors.
  • ArgoCD: GitOps continuous delivery tool for Kubernetes. Syncs cluster state with Git repositories.

Deployment Strategies

  • Rolling Update: Gradually replace old instances with new ones. Zero downtime but both versions run simultaneously during the update.
  • Blue-Green: Maintain two identical environments. Switch traffic from blue (current) to green (new) once validated. Enables instant rollback.
  • Canary: Route a small percentage of traffic to the new version. Gradually increase if no issues are detected. Minimizes blast radius of failures.
  • A/B Testing: Route traffic to different versions based on specific rules (location, user group) to test features with real users.

Module 7: Infrastructure as Code

What Is Infrastructure as Code?

Infrastructure as Code (IaC) is the practice of managing and provisioning infrastructure through machine-readable definition files rather than manual processes or interactive configuration tools. IaC enables version control for infrastructure, repeatable deployments, and automated testing of infrastructure changes.

Terraform

Terraform by HashiCorp is the most popular open-source IaC tool. It uses a declarative language called HCL (HashiCorp Configuration Language) to define infrastructure across multiple cloud providers. Terraform tracks the state of your infrastructure and only applies the changes needed to reach the desired state.

  • Providers: Plugins that interact with cloud platforms (AWS, Azure, GCP, Kubernetes, etc.)
  • Resources: The infrastructure components you create (instances, networks, databases)
  • State: Terraform tracks the current state of infrastructure in a state file
  • Modules: Reusable, shareable packages of Terraform configuration
  • Workflow: terraform init → terraform plan → terraform apply → terraform destroy

AWS CloudFormation

CloudFormation is AWS's native IaC service. It uses JSON or YAML templates to define AWS resources as "stacks." While CloudFormation only works with AWS, it has deep integration with all AWS services and is often the first IaC tool AWS-focused teams learn.

Ansible

Ansible is an open-source automation tool primarily used for configuration management, application deployment, and orchestration. Unlike Terraform (which provisions infrastructure), Ansible excels at configuring the software and settings on existing infrastructure. It is agentless, using SSH to connect to servers.

  • Playbooks define tasks in YAML format
  • Inventory files list managed hosts and groups
  • Roles organize playbooks into reusable components
  • Ansible Galaxy provides pre-built community roles

IaC Best Practices

  • Store all IaC in version control (Git) and review changes through pull requests
  • Use remote state storage (S3, Azure Blob, GCS) with state locking for team collaboration
  • Organize code into reusable modules to avoid duplication
  • Use separate workspaces or directories for different environments (dev, staging, prod)
  • Never hardcode secrets in IaC files; use secret management tools
  • Run terraform plan in CI pipelines and require approval before apply

Module 8: Monitoring & Logging

Why Monitoring Matters

Monitoring and observability are essential for understanding the health, performance, and behavior of your systems in production. Without proper monitoring, you are flying blind—unable to detect issues before they impact users, unable to diagnose root causes efficiently, and unable to make data-driven decisions about capacity and performance.

The Three Pillars of Observability

  • Metrics: Numerical measurements collected over time (CPU usage, request latency, error rates, throughput). Best for alerting and dashboards.
  • Logs: Timestamped text records of discrete events (application errors, access logs, system events). Best for debugging specific issues.
  • Traces: Records of the path a request takes through a distributed system. Essential for debugging microservices architectures.

Prometheus

Prometheus is an open-source monitoring system designed for reliability and scalability. It collects metrics from configured targets at given intervals, evaluates rule expressions, displays results, and triggers alerts when conditions are met. Prometheus uses a pull-based model and stores time-series data efficiently.

  • PromQL query language for flexible data analysis
  • Service discovery for automatically finding targets to monitor
  • Alertmanager for routing, deduplicating, and silencing alerts
  • Widely adopted as the standard for Kubernetes monitoring

Grafana

Grafana is the leading open-source platform for data visualization and dashboarding. It connects to dozens of data sources including Prometheus, Elasticsearch, CloudWatch, and Azure Monitor. Grafana enables teams to create rich, interactive dashboards to visualize metrics, logs, and traces in a single pane of glass.

ELK Stack (Elasticsearch, Logstash, Kibana)

The ELK Stack is the most popular open-source log management solution. Elasticsearch stores and indexes log data for fast search. Logstash (or Fluentd/Fluent Bit) collects, transforms, and ships logs. Kibana provides a web interface for searching, analyzing, and visualizing log data. Many teams now use the EFK stack (replacing Logstash with Fluentd) for Kubernetes environments.

Cloud-Native Monitoring

  • AWS CloudWatch: Monitoring service for AWS resources with metrics, logs, alarms, and dashboards
  • Azure Monitor: Full-stack monitoring for Azure including Application Insights for application performance
  • Google Cloud Operations (formerly Stackdriver): Integrated monitoring, logging, and tracing for GCP

Monitoring Best Practices

  • Monitor the four golden signals: latency, traffic, errors, and saturation
  • Set up alerts based on symptoms (user impact), not just causes (high CPU)
  • Use structured logging (JSON format) for easier parsing and analysis
  • Implement centralized log aggregation; never rely on SSH-ing into servers to read logs
  • Create runbooks for common alert scenarios to speed up incident response
  • Practice chaos engineering to validate monitoring and alerting before real failures occur
